01 · The thesis

An AI support agent that can't break policy.

Ava handles e-commerce refunds over chat, in-browser voice, and a real phone line — but the model never decides eligibility. A deterministic engine does. The LLM communicates; the engine is the authority.

04 · The rules

The rules make the call, not the AI

lib/policy/engine.ts
// no AI in the decision — order in, decision + audit trail out
export function evaluateRefund(order, _reason, _history): PolicyEvaluation {
  rules.push({ rule: "delivered",            passed: delivered });
  rules.push({ rule: "not_already_refunded", passed: notRefunded });
  rules.push({ rule: "return_window",        passed: inWindow });   // 30 days
  rules.push({ rule: "item_eligibility",     passed: anyEligible }); // partial = filter

  const hardDeny = rules.filter((r) => !r.passed);
  return hardDeny.length
    ? { decision: "deny",    eligibleAmount, rules, summary }
    : { decision: "approve", eligibleAmount, rules, summary };
}
View engine.ts on GitHub ↗
  • A fixed set of rules decides every refund — not the AI
  • Like: is it within 30 days? was it already refunded?
  • Same order always gets the same answer — no guessing
05 · The safety net

It can't refund what it shouldn't

lib/tools/index.ts · issue_refund
// Final deterministic gate — the LLM cannot bypass policy here.
const evaluation = evaluateRefund(evalOrder, reason, history);
if (evaluation.decision !== "approve") {
  return { ok: false, refused: true, message: "Refund refused by policy engine…" };
}
if (Math.abs(amount - evaluation.eligibleAmount) > 0.01) {
  return { ok: false, refused: true, message: "Amount does not match eligible…" };
}
// …only now does money move.
View tools/index.ts on GitHub ↗
  • The AI talks to the customer, but it can't approve a refund on its own
  • Before any money moves, the refund gets checked against the rules
  • If the rules say no, it won't pay out — even if someone tries to talk it into it
